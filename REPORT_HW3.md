# Отчёт по настройке CarbonPHP для Claude Code CLI

## 1. Описание AI-инструмента

**Инструмент:** Claude Code CLI (Command Line Interface)

**Версия:** 2.x+

**Назначение:** Claude Code - это инструмент командной строки от Anthropic, который позволяет запускать автономных AI-агентов для помощи в разработке. Агенты могут читать файлы, создавать и редактировать код, запускать команды в терминале, использовать внешние MCP-инструменты и даже создавать под-агентов для параллельного выполнения задач.

**Основные возможности:**
- Автономная работа с кодом без IDE
- Интерактивный REPL режим для диалогового взаимодействия
- Print mode для разовых запросов (удобно для скриптов)
- Система разрешений для контроля доступа
- Поддержка пользовательских команд и навыков
- Интеграция с MCP серверами для расширения возможностей
- Система памяти проекта через CLAUDE.md файлы
- Автоматические хуки для форматирования и проверок

**Требования:**
- Node.js 18+ (для установки через npm)
- Claude Pro/Max подписка или Anthropic API key
- PHP 8.1+ (для проекта CarbonPHP)
- Composer (для управления зависимостями)

**Установка:**
```bash
npm install -g @anthropic-ai/claude-code

# Настройка API ключа
export ANTHROPIC_API_KEY="your-api-key-here"

# Или авторизация через браузер (для Pro/Max)
claude auth
```

**Механизм работы с инструкциями:**

Claude Code использует иерархическую систему конфигурации:

1. **CLAUDE.md файлы** - содержат контекст и инструкции:
   - `~/.claude/CLAUDE.md` - глобальные инструкции для всех проектов
   - `.claude/CLAUDE.md` - инструкции уровня проекта
   - Вложенные `.claude/CLAUDE.md` - специфичные для директорий
   - Приоритет: более специфичные (вложенные) переопределяют общие

2. **settings.json файлы** - настройки разрешений и поведения:
   - `~/.claude/settings.json` - пользовательские настройки
   - `.claude/settings.json` - настройки проекта
   - Схема приоритетов: Managed > Project > User

3. **Skills** (навыки) - специализированные гайды для задач:
   - Markdown-файлы с инструкциями для конкретных задач
   - Claude сам решает когда их применять
   - Можно вызывать через natural language

4. **Custom Commands** - пользовательские slash-команды:
   - Хранятся в `.claude/commands/`
   - Вызываются через `/command-name`

**Преимущества для CarbonPHP:**
- Автоматическое форматирование кода через PHP-CS-Fixer
- Статический анализ через PHPStan после каждого изменения
- Контроль доступа к критичным файлам (composer.json, .github/, LICENSE)
- Удобная работа с тестами и локализацией
- Возможность быстро итерировать над изменениями

## 2. Описание сделанного

### Созданные файлы

#### 2.1. `.claude/CLAUDE.md` - Основной файл инструкций

**Содержание:**

1. **Назначение проекта** (раздел 1)
   - Краткое описание Carbon как библиотеки для работы с датами
   - Основные сценарии использования (создание, манипуляция, форматирование)
   - Инструкции по запуску проекта (composer install, phpunit, etc.)

2. **Структура проекта** (раздел 2)
   - Полная схема директорий с описанием
   - Ответственность каждого модуля (Carbon.php, Traits/, Lang/, tests/)
   - Объяснение разделения на mutable и immutable версии

3. **Ключевые термины** (раздел 3)
   - Мини-глоссарий специфичных терминов проекта
   - Carbon, CarbonImmutable, CarbonInterval, CarbonPeriod
   - Fluent Interface, Locale, Timezone, Macro, TestNow

4. **Ограничения** (раздел 4)
   - **Что НЕЛЬЗЯ менять:**
     - Публичный API (обратная совместимость критична)
     - Формат файлов локализации
     - Базовое поведение PHP DateTime
     - Структура composer.json
   - **Что можно менять:**
     - Внутренняя реализация
     - Добавление новых методов
     - Оптимизация и тесты

5. **Требования к вопросам и допущениям** (раздел 5)
   - Когда Claude ДОЛЖЕН спрашивать:
     - Неоднозначные требования
     - Изменения публичного API
     - Выбор между решениями
     - Работа с локализацией
   - Когда Claude МОЖЕТ делать допущения:
     - Стандарты кодирования (PSR-12)
     - Паттерны тестирования
     - Внутренняя реализация
     - Документация

6. **Требования к результату** (раздел 6)
   - **Стиль кода:**
     - PSR-12 обязателен
     - Type hints для всех параметров и возвратов
     - PHPDoc комментарии
     - Правила именования (camelCase, PascalCase, UPPER_CASE)
   - **Требования к тестам:**
     - 90%+ покрытие кода
     - PHPUnit с data providers
     - Тестирование граничных случаев
   - **Формат ответа:**
     1. План действий (перед началом)
     2. Изменения в коде (с объяснениями)
     3. Тесты
     4. Шаги проверки
     5. Проверочное требование (специальный комментарий)

7. **Примеры типовых задач** (раздел 7)
   - Сценарий 1: Добавление нового метода форматирования
   - Сценарий 2: Исправление бага
   - Сценарий 3: Оптимизация производительности
   - Каждый сценарий с примером промпта и ожидаемым ответом

8. **Критерии готовности** (раздел 8)
   - Чек-лист для Definition of Done
   - Все проверки должны пройти
   - Обратная совместимость сохранена
   - Документация обновлена

9. **Проверочное требование** (раздел 6.5)
   - В начале каждого PHP файла должен быть специальный комментарий:
     ```php
     /**
      * This file is part of the Carbon package.
      * Modified by Claude Code CLI
      */
     ```
   - Это демонстрирует, что Claude следует всем инструкциям!

**Особенности:**
- Подробное объяснение архитектуры проекта
- Четкие границы того, что можно и нельзя менять
- Конкретные примеры правильного кода
- Проверочное требование для демонстрации соблюдения инструкций
- Акцент на обратную совместимость (критично для библиотек)

#### 2.2. `.claude/settings.json` - Файл настроек

**Содержание:**

1. **Permissions (разрешения):**
   - **allow (разрешено):**
     - Composer команды (install, update, require)
     - Инструменты проверки (phpunit, php-cs-fixer, phpstan)
     - PHP команды для проверки синтаксиса
     - Git команды для просмотра (status, diff, log)
     - Чтение всех исходников (src/, tests/)
     - Редактирование исходников (src/, tests/)
   
   - **deny (запрещено):**
     - Опасные команды (rm -rf, sudo)
     - Сетевые команды (curl, wget)
     - Git push/commit (должен делать человек)
     - Доступ к .env файлам
     - Изменение конфигураций (composer.json, phpunit.xml.dist)
     - Изменение .github/ и LICENSE

2. **Hooks (хуки после выполнения команд):**
   - **php-cs-fixer-auto-format:**
     - Автоматически форматирует PHP файлы после редактирования
     - Запускается только для *.php файлов
     - Использует --quiet для минимального вывода
   
   - **phpstan-check:**
     - Запускает статический анализ на измененных файлах в src/
     - Помогает находить ошибки сразу
     - Не блокирует выполнение (|| true)

3. **Sandbox (песочница):**
   - Отключена для упрощения работы
   - Исключены docker, podman, kubectl

4. **Company Announcements (напоминания):**
   - Напоминание о проверочном комментарии
   - Напоминание о запуске тестов
   - Напоминание о стиле кода PSR-12

**Преимущества настроек:**
- Безопасность: нельзя случайно удалить важные файлы
- Автоматизация: код форматируется и проверяется автоматически
- Контроль: критичные файлы защищены от изменений
- Удобство: не нужно вручную запускать форматтеры

## 3. Трудности при выполнении задания

### 3.1. Изучение структуры проекта

**Проблема:** CarbonPHP - это большой проект с 200+ файлами локализации и сложной архитектурой на трейтах.

**Решение:** 
- Сфокусировался на основных классах (Carbon, CarbonImmutable)
- Изучил структуру трейтов (Creator, Difference, Comparison, etc.)
- Понял систему локализации через папку Lang/
- Проанализировал существующие тесты для понимания паттернов

### 3.2. Баланс между подробностью и лаконичностью

**Проблема:** Нужно было описать проект достаточно подробно для AI, но не перегрузить его информацией.

**Решение:**
- Структурировал CLAUDE.md по логическим блокам
- Добавил конкретные примеры кода вместо абстрактных описаний
- Использовал проверочное требование (специальный комментарий) для явной демонстрации
- Добавил 3 сценария типовых задач с ожидаемым форматом ответа

### 3.3. Ограничения на изменения

**Проблема:** Carbon - это production библиотека, используемая в миллионах проектов. Нужно было четко указать, что можно менять, а что нельзя.

**Решение:**
- Создал раздел "Ограничения" с явным разделением на "что нельзя" и "что можно"
- Объяснил причины каждого ограничения (обратная совместимость, формат локализации)
- Добавил правило о необходимости уточняющих вопросов при работе с публичным API

### 3.4. Настройка permissions в settings.json

**Проблема:** Нужно было найти баланс между безопасностью и удобством работы.

**Решение:**
- Разрешил все необходимые команды для разработки (composer, phpunit, php-cs-fixer)
- Запретил опасные команды (rm -rf, sudo) и сетевые запросы
- Защитил критичные файлы (composer.json, LICENSE, .github/)
- Добавил хуки для автоматического форматирования и проверки

### 3.5. Проверочное требование

**Проблема:** Нужно было придумать способ явно продемонстрировать, что Claude следует инструкциям.

**Решение:**
- Добавил требование о специальном комментарии в начале каждого файла
- Комментарий включает фразу "Modified by Claude Code CLI"
- Это легко проверяется визуально
- Отсутствие комментария = Claude не прочитал инструкции

### 3.6. Формат ответа Claude

**Проблема:** Нужно было стандартизировать ответы Claude для консистентности.

**Решение:**
- Создал четкую структуру: План → Изменения → Тесты → Проверка
- Добавил примеры для каждого типа задач
- Требовал объяснение каждого решения с альтернативами
- Указал на необходимость команд для проверки
